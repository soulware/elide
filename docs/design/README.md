explorations. Each file covers one topic.

Per-tick cap (`max_buckets_per_tick`, default 4) is sized against retention-peak |

| [portable-live-volume.md](portable-live-volume.md) | Accepted: named volumes become portable across hosts; each `volume start` is a fresh fork inheriting from the previous tail, so each ownership episode has its own ULID and signing key. The only shared mutable thing is `names/<name>` |
| [volume-size-ownership.md](volume-size-ownership.md) | Implemented: `size` lives on the `names/<name>` claim record (single owner, CAS-protected, signed in event log), not on the unsigned `manifest.toml`. Ancestors carry no size; resize is a CAS + `UBLK_U_CMD_UPDATE_SIZE` (Linux 6.16+) |
| [manifest-toml-removal.md](manifest-toml-removal.md) | Implemented: dropped `manifest.toml` entirely. `name` / `readonly` / `origin` were redundant with existing surfaces; OCI `source` migrated onto signed `volume.provenance` as `oci_source`. Fresh-bucket-only |
| [volume-event-log.md](volume-event-log.md) | Proposed: per-name append-only event log under `events/<name>/<ulid>` recording lifecycle transitions (claim, release, force-release, start/stop, fork, rename). Pointer stays canonical for "now"; log is canonical for "ever". Includes a rename design that ties two name logs together without copying history |
| [segment-index.md](segment-index.md) | Proposed: P3 (folding P4) of list elimination — replace the per-vol `segments/`+`retention/` LISTs with a *second manifest* at `by_id/<vol>/HEAD` (single overwritten object; named like `events/<name>/HEAD` for the same "leading edge of activity" role) for the post-snapshot delta over P2's `LATEST` anchor. Reaper folded into the per-volume tick loop → sole sequential writer; HEAD PUT per drain tick on any state change, one GET reads any-size HEAD, no lock/chain. Cross-coordinator only (owner uses local `index/`); derived/unsigned over a per-segment-signed substrate |
| [force-release-fencing.md](force-release-fencing.md) | Proposed: split-brain safety for `volume claim --force`. The claimant's basis is the owner's own published snapshot, already pinned by the owner's GC floor; the fence is the `volume-rw` liveness predicate (forced CAS kills the zombie's credential renewals); the tail re-own copy retries through HEAD re-resolution |
| [read-state-divergence-check.md](read-state-divergence-check.md) | Proposed: GC-commit enforcement of the single-writer rule for a volume's own layer. The daemon keeps an own-layer segment set (mirroring what a fresh `open_read_state` would load, maintained at the same IPC points that mutate local read form) and requires `plan.inputs ⊆ own_segments` in `apply_plan_apply_result` — closing the stale-liveness check's blind spot (unknown-input hashes fail `still_at_input` and are skipped, which is how the 2026-07-02 divergence got folded). On violation: ERROR + daemon exits; supervisor respawn rebuilds from disk and converges (recovery *is* the rebuild). Rejected inputs are never Superseded, so the reaper can't delete the unseen bytes. Detection is at GC cadence — it caps the divergence window, prevention stays with the claim gate |
| [claim-supervision-gate.md](claim-supervision-gate.md) | Implemented: `volume.claiming` marker (parallel to `volume.importing`) making a mid-claim fork dir structurally invisible to `discover_volumes` until `finalize` parks it. Closes a live-repro'd data-loss race in `claim --force`: discovery admits `volume.key`-only dirs (34e0d9c), so the daemon could open the fork after `rebind` wrote provenance but before `re_own` landed the head delta — serving the pre-delta state, which GC durably folded in. Plain claim shares the window but is content-complete-by-reference at the handoff pin, so it only crash-looped or started early |
| [displaced-fork-rehome.md](displaced-fork-rehome.md) | Proposed: the *previous owner's* local disposition once `names/<name>` binds a different fork (the "hygiene" side `force-release-fencing.md` defers). A *rehomes* its fork under `<name>-<suffix>` (six hex chars, hash-derived from the episode so the probe is crash-idempotent with no persisted state) — the name carries no information; the disposition lives in an event on the new name's log: `superseded` when A cleanly released and a peer claimed (a drained predecessor, lineage note), `displaced` when a peer force-claimed while A still held it (a divergent branch; A fences the guest to a stop with an honest `EIO`; the event cross-links the claimant). The `volume.released` marker discriminates. Both dispositions land the fork `released` under its new name (record minted `Released` + marker), so recovery is uniformly reclaim-then-start. Rehome is fork-rehoming not name-renaming (the old name stays live under the claimant, so `volume-event-log.md`'s two-event rename boundary doesn't apply). Displacement uniquely preserves the undrained tail (reclaim → start → re-attests → drains). One `rehome(fork)` primitive fires at three sites (running poll, start-refusal, `force_claim` finalize). Preserve-by-default; removal is the ordinary local `volume remove` (bucket-level deletion is unsupported for any volume, so out of scope) |
| [ancestor-liveness.md](ancestor-liveness.md) | Proposed: `by_id/<ulid>/` dirs are shared state (a named volume's home *and* other volumes' ancestor layers), but `volume remove` decides from the target's own lifecycle alone — live-repro'd on coord-1 (2026-07-03): removing rehomed `vol2-2595e8` deleted the new vol2's frontier parent and crash-looped its daemon at `verify_ancestor_manifests`. Design: one reachability definition (anchors = `volume.key` / name binding / in-flight markers; skeletons = `volume.readonly` dirs, live iff some anchor's `lineage_ulids` reaches them — "root" stays reserved for lineage topology) enforced both ways in the tick loop — remove demotes to the pull-skeleton shape instead of deleting, a sweep deletes unreachable skeletons, a heal pass re-runs the `prefetch_indexes` chain walk on lineage incompleteness (new trigger; today it fires only on the freshly-pulled-fork heuristic, so a populated fork with a missing ancestor crash-loops instead of self-repairing). Bodies are already demand-fetched; the unit of required local presence is the ancestor's metadata read form (provenance, manifest, `.idx` sections) at open. A local-only `volume tree` renders the same lineage forest (anchors vs skeletons, snapshot-pinned fork edges, missing ancestors as heal candidates) — the low-risk first consumer of the computation before it is wired to deletion |
| [consistency-surface.md](consistency-surface.md) | Exploration: which Elide operations require strong consistency vs. which tolerate eventual; failure-mode walkthrough; sketch of a two-bucket split (small strongly-consistent coordination bucket, large eventually-consistent data bucket) |
| [tigris-native.md](tigris-native.md) | Exploration: what Elide looks like if designed Tigris-native (bucket snapshots, forks, versioning as first-class primitives) rather than as a portable S3 consumer |
| [iam-key-model.md](iam-key-model.md) | Superseded by docs/design/mint.md (coordinator in-process path + `[iam]` section removed); retained for the key-inventory/policy-scoping rationale mint's roles inherit. Per-volume IAM key model for Tigris-style backends. Four key classes (admin, one writer per coordinator, one peer-fetch per coordinator, per-volume RO with ancestor-inclusive policies). IAM-layer invariants: `events/` is append-only and `coordinators/` is immutable (no key holds Delete). Identity via policy names (Tigris doesn't tag keys); host-local reconciliation; dead-host orphans need operator action |
| [auth-model.md](auth-model.md) | Auth model and principles: isolation guarantees on a shared-uid host (what macaroons enforce and don't); the settled principle that operator authorisation gates S3 *writes* not destructive verbs. Operator IPC verbs are ungated in the codebase today; the concrete central-auth-service design lives in docs/design/auth-service.md. References `architecture.md` for the keyed-BLAKE3 macaroon construction shared with volume macaroons |
| [auth-service.md](auth-service.md) | Proposed: central auth service issues per-op discharges; mint is the sole primary-macaroon issuer. HMAC throughout (keyed-BLAKE3, same construction as volume macaroons) — the auth service is a third-party authority discharging a TPC embedded in mint-issued primaries. Multi-tenant by construction (mandatory `OrgId`; mint is per-org and the org-identity broker; coord enrolls to a mint, not direct to auth service). Operator login produces a session discharge; CLI trades it for per-op discharges at IPC time. Mint-as-auth packaging for dev/test/demo via `demo-enabled = true`. Concrete `/v1/` API surface for login, discharge, mint enrollment, and coord enrollment. Clean-break migration from PoC |
| [remote-coord-ipc.md](remote-coord-ipc.md) | Exploration: running operator IPC verbs (`volume snapshot`, etc.) against a coord on another host. Decision: route `cli ↔ local coord ↔ remote coord` rather than `cli ↔ remote coord` direct — keeps the CLI unauthenticated, places the operator session at coord (consistent with coord-as-clearer + per-forward attenuation in docs/design/auth-service.md), and reuses the existing peer channel from docs/design/peer-segment-fetch.md. Assumes a coord is always running on the caller's host |
| [coord-http-ipc.md](coord-http-ipc.md) | Proposed: pivot coord inbound socket (CLI ↔ coord) from NDJSON+UDS to HTTP+UDS, matching mint and the auth-service. RPC-style POST /v1/<verb>, JSON bodies, HTTP status codes replace `IpcErrorKind`, `Authorization: Macaroon <bundle>`, `401 + WWW-Authenticate: Macaroon` for the discharge challenge, chunked NDJSON for streaming verbs. Volume control socket (coord ↔ volume) is out of scope and stays NDJSON+UDS |
| [mint.md](mint.md) | Elide's **consumption view** of `mint`, now a separate OSS project ([github.com/soulware/mint](https://github.com/soulware/mint)) with its own design docs. Covers the four-role inventory the coordinator enrolls for and assumes (`coord-ro`, `coord-rw`, `volume-rw`, `volume-ro`), the coordinator-side credential plumbing (`ScopedStores`, keypair cache), the `[mint]` config, and the always-attest caveat-provenance model as Elide authors it (issuer-stamped reserved caveats vs attested non-reserved ones; literal deployment constants). Mint's own internals — protocol, keyring, template seal, enrollment / `assume-role` / `exchange-finalize` endpoints — live in the mint repo; the sealed role inventory Elide runs mint with is `deploy/mint/` |
| [mint-template-seal.md](mint-template-seal.md) | Redirect: the template-seal mechanism (signed `seal.json`, dormant-until-sealed startup, `POST /v1/admin/seal`, sealed cache) is **mint's**, owned by the mint repo ([github.com/soulware/mint](https://github.com/soulware/mint)). Elide owns the sealed *input* — the role policy templates and `mint-elide.toml` in `deploy/mint/`, sealed by `mint seal` at deploy time |
| [mint-volume-attestation.md](mint-volume-attestation.md) | Implemented: mint's per-volume scoping value `volume` is **attested** — a non-reserved `{{caveat.volume}}` vouched by an attestation coordinator's discharge — rather than self-asserted, closing the gap where a coordinator could request RW credentials to any volume's prefix. A third-party caveat in the credential is discharged by coord B, co-located with mint; coord A proves possession of the live volume's `volume.key` against the public `meta/<vol>.pub`, and coord B derives the read set (self RW + per-ancestor RO) from the signed `meta/<vol>.provenance` lineage and `names/<name>` currency. coord B is a pure function over public signed state (holds no secret); mint stays volume-agnostic and binds the principal via `cnf`. Generalises to the ancestor chain with one possession proof anchored at the live volume |

| [peer-segment-fetch.md](peer-segment-fetch.md) | Exploration: opportunistic LAN peer-fetch tier in front of S3 for index/body bytes. Targets cross-host handoff (release → claim) and large-fleet image pull. URL space mirrors S3 paths; auth mirrors per-volume IAM prefix scope. |
| [deployment-modes.md](deployment-modes.md) | Proposed: three deployment modes (`coord run`, `coord start`/`stop`, systemd) with a uniform operator surface. Internal `VolumeLauncher` trait abstracts direct-fork from systemd transient units (`elide-vol-<ulid>.service`); each volume in its own cgroup so default `KillMode=control-group` on the coord unit lets volumes outlive coordinator restart. Avoids the `KillMode=process` path explicitly discouraged in `systemd.kill(5)` |
| [deploy-image-pipeline.md](deploy-image-pipeline.md) | Build the mint + elide binaries in GitHub Actions and publish them as release artifacts (public repos → tokenless download); the deploy Dockerfile `curl`s the pinned version in place of `git clone … && cargo build`, and the build stage runs `mint render` with the downloaded binary. Both repos publish and both deploy images consume by download: `deploy/mint/` downloads and checksum-verifies the released mint binary at a pinned `MINT_VERSION` (mint #43, first tag `v0.1.0`), and elide's `release.yml` publishes its three triple-named binaries (`elide`, `elide-coordinator`, `elide-import`) with the tag baked into `--version` via elide-core's `build.rs` (first tag `v0.1.0`); `deploy/elide/` downloads and checksum-verifies all three at a pinned `ELIDE_VERSION`. Names the three failure modes of the source-build-in-Dockerfile model (silent stale-binary cache, slow cold builds, three-place cross-repo lockstep — empirically the pre-#41 `MINT_REF` that broke enrollment). Open: where the `(mint, elide)` version pin lives — elide's `deploy/` or a standalone deploy repo |
| [domain-store.md](domain-store.md) | Proposed: replace `Arc<dyn ObjectStore>` (215 occurrences across 23 files in `elide-coordinator/src/`) with object-typed handles vended by role — `NameClaims`, `EventJournal`, `OwnIdentity`, `VolumeData` (segments/snapshots/head/metadata sub-accessors), `ControlPlaneReader`. Migrates per-object alongside `ScopedStores`, no atomic swap |
