# Elide Status — 2026-06-29

Progress since [2026-06-24](status-2026-06-24.md).  Five days,
~17 PRs (#609–#625).

The deploy images stopped compiling from source and started
downloading released binaries; both repos pin a single version
knob; the full mint-backed stack went operational on real Fly;
and a no-mint `elide-standalone` deployment proved
cross-deployment volume handoff.

---

## Major changes

### Release-binary deploy pipeline

Both repos publish their binaries as GitHub Release artifacts and
both deploy images consume them by download instead of
`git clone … && cargo build` (proposal #608):

- elide's `release.yml` publishes three triple-named binaries
  (`elide`, `elide-coordinator`, `elide-import`) on `v*` tags,
  with the tag baked into `--version` via `build.rs`; `deploy/coord`
  downloads and checksum-verifies all three (#617/#618).
- `deploy/mint` pulls the released mint binary rather than
  compiling it (#613).
- `v0.1.0` and `v0.1.1` are tagged and released.

### One version knob per deployment

`ELIDE_VERSION` becomes the single elide-side deploy knob and
`MINT_VERSION` the single mint-side knob (#620–#622).  A
`deploy.sh` wrapper resolves the release version, preflights that
the release assets exist, and accepts an explicit `latest`
selector (#623).  `deploy/coord` is then renamed to `deploy/elide`
(#624).

### Coordinator self-attestation via shared K_M-B

The coordinator reads the attestation discharge route from the
caveat and self-attests with a shared `K_M-B`; the mint-side spec
for sharing `K_M-B` is written and `deploy/mint` shares it at the
pinned `v0.1.1` (#611/#612/#614).  The earlier socat demo-auth
bridge is removed (#615).

### Full stack operational on real Fly

The mint-backed deployment — mint-for-elide plus a coordinator,
attested per-volume credentials — came up end-to-end on real
Fly.io (2026-06-27), serving a real volume.  ublk was confirmed
to work on Fly: the module ships but is not loaded by default, so
the coordinator image loads `ublk_drv` at boot.

### elide-standalone and cross-deployment handoff

`deploy/elide-standalone` deploys a coordinator that talks to
Tigris directly with a read/write keypair — no mint, no
enrollment, no attestation — and vends that same keypair to each
volume (#625).

Cross-deployment volume handoff was proven on real Fly
(2026-06-29): a volume released on the mint-backed coordinator was
claimed on the standalone coordinator over the same shared bucket,
mounted, and its contents read back intact.  One-owner is
structural — a claim only proceeds from a released volume, and
`--force` is the sole override, still ETag-CAS fenced.

## Bug fixes

- **Credential lifetime** (#610) — the coordinator records the
  attenuated `exp` as the credential lifetime rather than the
  unattenuated parent's.
- **Volume liveness** (#616) — liveness is classified by the
  volume lock, not a bare pid, so a recycled pid no longer reads
  as a live volume.
- **ublk device ids** (#619) — device ids are stable across
  reboot; `--ublk-id` is dropped.

## Remaining work

The comment rule was sharpened into a positive litmus (#609) and
the CLAUDE.md guidance updated to match.

Known deploy gaps: a stopped-old-volume name-record case and an
empty-`/mnt` remount path remain to be ironed out, and the demo
shared-key configs are proven via an in-container shim but not yet
converged in the committed configs.

Carried-over priorities: boot hints + S3 repacking, snapshot
indexes, reclamation thresholds.  ublk zero-copy stays deferred,
though ublk itself now runs on Fly.
